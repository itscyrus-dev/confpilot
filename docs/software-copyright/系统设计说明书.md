# ConfigPilot 系统设计说明书

## 一、总体架构

ConfigPilot 采用前后端分层架构：

- 前端层：React、TypeScript、Vite，负责界面展示和用户交互。
- 桌面容器层：Tauri 2，负责窗口管理和前后端通信。
- 后端核心层：Rust，负责文件扫描、GitHub API、Git 仓库同步、系统安全存储、文件监听和冲突处理。
- 云端存储层：GitHub 私有仓库，负责保存配置文件版本历史。

## 二、模块设计

### 1. 配置扫描模块

扫描固定的 zsh 与 Ghostty 配置路径，判断文件或目录是否存在，计算哈希值，读取修改时间，并将结果返回给前端展示。

### 2. GitHub 授权模块

使用 GitHub Device Flow 完成授权。后端请求设备码，前端展示验证码，用户完成授权后后端轮询 access token，并通过系统 Keychain 保存 token。

### 3. 仓库管理模块

授权成功后，后端调用 GitHub API 检查 `configpilot-dotfiles` 仓库是否存在。若不存在，则创建私有仓库；若存在，则复用已有仓库。

### 4. 同步模块

同步模块将本地配置复制到应用数据目录中的 Git 工作区，生成 manifest 文件，提交并推送到 GitHub。恢复时从仓库工作区复制配置到原始路径，并在覆盖前生成备份。

### 5. 文件监听模块

使用 Rust notify 库监听配置文件变化。监听事件经过防抖处理后触发备份同步，避免连续保存造成频繁提交。

### 6. 冲突处理模块

通过 manifest 记录的哈希与本地、远端当前哈希比较，检测本地和远端是否同时发生变化。发生冲突时生成本地副本和远端副本，并由用户选择处理方式。

### 7. 状态存储模块

应用状态保存在 Tauri app data 目录，包括 GitHub 用户、仓库名、最近同步时间、最近提交、监听开关和日志信息。

## 三、数据流

1. 用户启动软件。
2. 软件加载本地状态并扫描配置路径。
3. 用户完成 GitHub 授权。
4. 软件创建或复用私有仓库。
5. 用户执行备份或监听器自动触发备份。
6. 后端复制配置到工作区，生成 manifest，提交并推送。
7. 前端刷新状态并展示同步日志。

## 四、安全设计

软件不采集 GitHub 密码，不同步 `.ssh` 等高风险目录。GitHub token 使用系统安全存储。配置冲突时不自动覆盖本地文件，优先保护用户数据。
