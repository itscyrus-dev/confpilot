# ConfigPilot 产品需求文档 PRD

## 1. 产品概述

ConfigPilot 是一款面向 macOS 桌面用户的个人配置文件同步管理工具。产品聚焦 zsh 与 Ghostty 等开发环境配置，通过 GitHub 授权、私有仓库备份、本地文件监听、恢复和冲突处理，帮助用户在多台设备之间安全地保存和迁移个人配置。

产品第一阶段定位为“低打扰、可控、安全”的配置同步助手：默认只同步明确列入白名单的配置文件，不同步 SSH、系统钥匙串、浏览器数据、Git 凭据等高风险内容；发生冲突时生成本地与远端副本，交由用户选择，不静默覆盖。

## 2. 背景与问题

开发者和高级终端用户通常会长期维护自己的 shell、终端、主题、快捷键与别名配置。这些文件体积小但价值高，迁移到新电脑或在多台 Mac 间保持一致时，常见问题包括：

- 手动拷贝容易遗漏隐藏文件和目录。
- dotfiles 仓库需要用户理解 Git、目录映射和冲突处理。
- 直接同步整个配置目录存在隐私和凭据泄露风险。
- 多设备同时修改后，用户很难判断应该保留哪一份。
- 后台自动同步如果缺少可见状态，会让用户担心配置被意外覆盖。

ConfigPilot 以桌面应用形式封装 GitHub 私有仓库同步流程，让用户通过清晰的授权、扫描、备份、恢复和冲突处理界面完成配置管理。

## 3. 目标用户

### 3.1 主要用户

- 使用 macOS 的开发者、设计工程师、运维工程师和技术创作者。
- 使用 zsh、Ghostty 等终端工具，并拥有个性化配置的用户。
- 希望把 dotfiles 放在私有 GitHub 仓库，但不想手动维护脚本的用户。

### 3.2 次要用户

- 刚换 Mac 或需要重装环境的用户。
- 有多台 Mac，需要在办公设备和个人设备之间同步基础终端配置的用户。
- 团队内部希望推荐一套安全同步工具给非 Git 专家的成员。

## 4. 产品目标

- 降低个人配置备份门槛：用户无需手动创建仓库、编写脚本或记忆 Git 命令。
- 提升配置迁移效率：在新设备上授权后可从云端恢复配置。
- 降低数据风险：只同步白名单配置，GitHub token 使用系统安全存储。
- 保持用户控制感：关键操作明确展示状态、路径、仓库和冲突。
- 为后续扩展打基础：同步文件清单、仓库结构和冲突记录可扩展到更多工具。

## 5. 成功指标

- 首次授权和仓库准备流程可在 3 分钟内完成。
- 用户能在首页直接判断 GitHub 授权、仓库、配置文件和最近同步状态。
- 同步失败或冲突发生时，界面给出可理解的原因和下一步操作。
- 用户不需要离开应用即可完成备份、恢复、双向同步和冲突选择。
- 默认同步范围中不包含高风险敏感目录。

## 6. 范围

### 6.1 第一版范围

- GitHub 浏览器 OAuth 登录。
- GitHub Device Flow 备用登录。
- 自动创建或复用私有仓库 `configpilot-dotfiles`。
- 扫描以下配置：
  - `~/.zshrc`
  - `~/.zprofile`
  - `~/.zshenv`
  - `~/.config/zsh/`
  - `~/.config/ghostty/config`
  - `~/.config/ghostty/`
- 本地 app data 工作区缓存 Git 仓库。
- 手动备份、从云端恢复、双向同步。
- 自动监听配置文件变化并触发备份。
- 冲突检测、冲突副本保留、用户选择本地或远端版本。
- 同步日志、应用数据目录、本地同步仓库路径展示。
- macOS 托盘常驻、关闭窗口后隐藏。

### 6.2 暂不包含

- 同步 SSH、Keychain、浏览器配置、Git 凭据等高风险数据。
- 多 Git 服务提供商。
- 复杂的三方合并编辑器。
- 文件级版本历史可视化。
- 团队共享配置模板。
- Windows 和 Linux 桌面支持。

## 7. 核心用户故事

- 作为新用户，我希望通过 GitHub 登录完成授权，并自动准备一个私有仓库，以便不需要手动配置 Git。
- 作为长期使用者，我希望打开首页就看到有哪些配置文件被发现、最近一次同步时间和监听状态。
- 作为有多台设备的用户，我希望能手动执行“立即备份”和“从云端恢复”，以便在换机时快速迁移配置。
- 作为谨慎用户，我希望发生本地和远端差异时，应用不要自动覆盖，而是让我选择使用哪一份。
- 作为后台使用者，我希望应用关闭窗口后仍能在托盘中运行，并继续监听配置变化。

## 8. 功能需求

### 8.1 首页总览

首页展示当前同步系统的核心健康状态：

- GitHub 授权状态：已授权时展示用户名，未授权时展示“未授权”。
- 仓库状态：展示 `owner/configpilot-dotfiles` 或默认仓库名。
- 配置文件发现数量：展示已发现数量与总清单数量。
- 最近同步时间：无记录时展示“尚未同步”。
- 自动监听：可刷新、开启监听、停止监听。
- 配置文件表格：展示配置名称、源路径、仓库路径和发现状态。

验收标准：

- 未授权用户进入应用后仍可查看本地配置扫描结果。
- 自动监听按钮根据当前状态切换为“开启监听”或“停止监听”。
- 表格长路径不撑破布局，支持截断或悬停查看完整路径。

### 8.2 GitHub 授权

授权页支持两种登录方式：

- 推荐方式：浏览器 OAuth 登录，应用打开 GitHub 授权页面，回调到本机 `127.0.0.1:39119/callback`。
- 备用方式：Device Flow，展示验证码并引导用户打开授权页面。

授权成功后：

- 保存 GitHub token。
- 获取 GitHub 用户名。
- 自动创建或复用 `configpilot-dotfiles` 私有仓库。
- 更新同步状态为已授权。

验收标准：

- 未授权时展示主按钮“GitHub 登录”和备用按钮“验证码备用登录”。
- 已授权时按钮进入禁用或已授权状态。
- 仓库准备失败时展示错误信息，不影响用户重新尝试。

### 8.3 配置扫描

应用扫描白名单内的文件和目录，并为每一项生成：

- 配置 ID。
- 展示名称。
- 本地源路径。
- 仓库目标路径。
- 是否存在。
- 是否目录。
- 文件 hash。
- 最近修改时间。

验收标准：

- 文件不存在时展示“未发现”，不阻塞其他文件同步。
- 目录存在时按目录递归处理。
- 扫描结果在首页、同步页保持一致。

### 8.4 手动同步

同步页提供三类操作：

- 立即备份：将本地白名单配置复制到本地仓库并推送到 GitHub。
- 从云端恢复：将远端仓库中的配置恢复到本地对应路径。
- 双向同步：拉取远端变化，与本地变化比较后同步。

验收标准：

- 未授权时同步按钮不可用。
- 同步进行中展示处理中状态，避免重复点击。
- 操作完成后刷新配置状态、最近同步时间、commit 信息和日志。

### 8.5 自动监听

用户可开启或停止自动监听。开启后，应用监听已存在的白名单文件和目录变化，并在变化后执行备份。

验收标准：

- 应用启动时如果 `auto_watch` 为 true，则自动启动监听。
- 文件变化触发需要有简单节流，避免短时间内重复提交。
- 停止监听后状态回到空闲。

### 8.6 冲突处理

冲突页展示本地和远端 hash 不一致的记录，包括：

- 原始源路径。
- 仓库目标路径。
- 本地冲突副本路径。
- 远端冲突副本路径。
- 本地 hash。
- 远端 hash。
- 冲突创建时间。

用户可选择：

- 使用本地。
- 使用远端。
- 打开源文件。

验收标准：

- 冲突发生时不静默覆盖用户配置。
- 冲突数量在页头明确展示。
- 冲突解决后刷新冲突列表。

### 8.7 设置与日志

设置页展示：

- GitHub 用户。
- 仓库。
- 应用数据目录。
- 本地同步仓库。
- 刷新状态。
- 打开数据目录。
- 清理本地缓存。
- 同步日志。

验收标准：

- 路径内容支持长文本换行。
- 清理本地缓存操作完成后给出反馈并刷新状态。
- 日志保持最近关键事件，便于用户排查问题。

## 9. 非功能需求

### 9.1 安全

- GitHub token 优先保存到系统安全存储，macOS 使用 Keychain。
- 本地状态文件不保存明文敏感信息。
- 默认不同步高风险文件和目录。
- 私有仓库作为默认仓库类型。

### 9.2 可靠性

- 所有同步操作需要捕获错误并回传到前端。
- 本地仓库目录不存在时自动创建。
- OAuth 回调端口被占用时给出明确错误。
- 网络失败、GitHub API 失败、Git 操作失败均应进入可恢复状态。

### 9.3 可用性

- 主导航固定在左侧，常用操作在当前页面顶部可见。
- 所有危险操作使用明确文案。
- 状态文案使用用户能理解的中文，不暴露过多底层异常术语。

### 9.4 性能

- 首页状态刷新应在正常环境下 1 秒内返回。
- 配置扫描范围保持白名单，避免递归扫描大量无关目录。
- 自动监听需要节流，避免频繁写入和推送。

## 10. 信息架构

- 首页：状态总览、自动监听、配置清单。
- 授权：GitHub 登录、Device Flow、仓库准备。
- 同步：备份、恢复、双向同步、配置清单、同步日志。
- 冲突：冲突列表、hash 对比、解决操作。
- 设置：账户、仓库、路径、缓存、日志。

## 11. 关键流程

### 11.1 首次使用流程

1. 用户打开 ConfigPilot。
2. 首页展示本地配置扫描结果和未授权状态。
3. 用户进入授权页并点击 GitHub 登录。
4. 应用打开 GitHub 授权页面。
5. 用户授权后回到应用。
6. 应用创建或复用私有仓库。
7. 用户进入同步页点击立即备份。
8. 首页更新最近同步时间和仓库状态。

### 11.2 换机恢复流程

1. 用户在新 Mac 安装并打开 ConfigPilot。
2. 用户完成 GitHub 授权。
3. 应用找到已有 `configpilot-dotfiles` 仓库。
4. 用户进入同步页点击从云端恢复。
5. 应用将远端配置写回本地路径。
6. 用户根据日志确认恢复结果。

### 11.3 冲突解决流程

1. 应用检测到本地和远端同一配置项 hash 不一致。
2. 应用生成本地和远端冲突副本。
3. 冲突页展示冲突记录。
4. 用户查看路径和 hash，必要时打开文件。
5. 用户选择使用本地或使用远端。
6. 应用应用选择结果并刷新同步状态。

## 12. 数据与状态

### 12.1 本地持久化状态

- `github_username`
- `repo_owner`
- `repo_name`
- `repo_private`
- `last_sync_at`
- `last_commit`
- `auto_watch`
- `sync_status`
- `logs`

### 12.2 前端运行态

- 当前 tab。
- busy 状态。
- 消息提示。
- Device Flow 信息。
- 配置文件列表。
- 冲突列表。
- 应用数据目录和本地仓库目录。

### 12.3 仓库结构

```text
zsh/
  .zshrc
  .zprofile
  .zshenv
  config-zsh/
ghostty/
  config
  ghostty/
manifest.json
```

## 13. 风险与应对

- GitHub OAuth 配置错误：在 README 和授权错误中提示 callback URL 与环境变量要求。
- 用户文件不存在：允许部分文件缺失，界面逐项展示。
- 冲突误覆盖：默认保留冲突副本，并由用户手动选择。
- 自动监听频繁提交：设置变更节流和日志提示。
- token 存储失败：降级到本地 fallback 时需要明确提示风险，后续版本可补充。

## 14. 后续版本方向

- 支持更多配置类型，例如 Starship、Git config、Neovim、tmux。
- 提供配置项开关，让用户选择同步范围。
- 增加版本历史和 diff 预览。
- 支持多仓库或自定义仓库名。
- 支持同步前 dry-run 预览。
- 支持 Windows 和 Linux。
